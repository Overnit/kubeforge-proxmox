# Homelab Proxmox — AI Instructions

## Project Context

This is an Infrastructure as Code (IaC) repository for provisioning a k3s Kubernetes cluster on Proxmox VE using Terraform. The Proxmox host is accessed through a Cloudflare Access tunnel via a local Python reverse proxy.

## Tech Stack

- **Terraform** with the `bpg/proxmox` provider (~> 0.78)
- **Proxmox VE 9.1.1** as the hypervisor
- **Ubuntu 24.04 LTS** cloud images with cloud-init for VM provisioning
- **k3s** for lightweight Kubernetes
- **Cloudflare Access** (Zero Trust) protecting the Proxmox API
- **Python 3** reverse proxy for injecting CF Access service token headers

## Repository Layout

- `terraform/` — All Terraform configuration files
  - `versions.tf` — Provider requirements (bpg/proxmox, hashicorp/tls, hashicorp/random)
  - `main.tf` — Provider configuration (endpoint, api_token, insecure=true)
  - `variables.tf` — All input variables with sensible defaults
  - `cloud-init.tf` — SSH key generation (ED25519) and Ubuntu cloud image download
  - `k3s-cluster.tf` — VM resources for master and worker nodes
  - `outputs.tf` — Master IP, worker IPs, SSH private key
  - `terraform.tfvars.example` — Template for user-specific values
  - `terraform.tfvars` — Actual secrets (NEVER committed, git-ignored)
- `scripts/cf-access-proxy.py` — HTTPS reverse proxy that injects CF-Access-Client-Id/Secret headers
- `scripts/setup-k3s.sh` — Bash script to install k3s server on master and agents on workers

## Conventions

- All sensitive values go in `terraform.tfvars` which is git-ignored.
- The proxy runs on `https://localhost:18006` and Terraform connects to it (not directly to Proxmox).
- VMs use static IPs assigned via cloud-init (no DHCP).
- SSH keys are auto-generated by Terraform (`tls_private_key`) — extract with `terraform output -raw k3s_ssh_private_key`.
- VM disks use `import_from` (not `file_id`) to import the cloud image. The `disk[0].import_from` is in `lifecycle.ignore_changes` to prevent unnecessary recreation.
- The cloud image is downloaded with `content_type = "import"` and renamed to `.qcow2` extension (Proxmox requires this for import).

## Important Patterns

- **CF Access Proxy**: Terraform cannot talk to Proxmox directly when it's behind Cloudflare Access. The Python proxy at `scripts/cf-access-proxy.py` intercepts HTTPS requests on localhost, injects the service token headers, and forwards to the real Proxmox host.
- **k3s Setup**: The `setup-k3s.sh` script reads Terraform outputs (IPs, SSH key) and SSHes into each VM to install k3s. It handles master first, retrieves the join token, then installs agents on workers.
- **VM Naming**: Master is `k3s-master-0`, workers are `k3s-worker-{index}`. VM IDs start at `var.k3s_vm_start_id` (default 200).

## Security Rules

- NEVER hardcode tokens, passwords, or secrets in `.tf` files.
- NEVER commit `terraform.tfvars`, `*.tfstate`, or `.terraform/` directory.
- All secrets must be passed via `terraform.tfvars` or environment variables.
- The CF Access proxy environment variables (`CF_ACCESS_CLIENT_ID`, `CF_ACCESS_CLIENT_SECRET`) must be set at runtime, never stored in code.
- The `proxmox_api_token` variable is marked `sensitive = true` in Terraform.

## Common Operations

```bash
# Start CF Access proxy (needed before any Terraform operation)
export CF_ACCESS_CLIENT_ID="<id>"
export CF_ACCESS_CLIENT_SECRET="<secret>"
python3 scripts/cf-access-proxy.py &

# Terraform workflow
cd terraform
terraform init
terraform plan
terraform apply

# Get SSH key for VMs
terraform output -raw k3s_ssh_private_key > /tmp/k3s-key && chmod 600 /tmp/k3s-key

# SSH into a node
ssh -i /tmp/k3s-key ubuntu@192.168.0.210

# Install k3s cluster
bash scripts/setup-k3s.sh /tmp/k3s-key

# Get kubeconfig
ssh -i /tmp/k3s-key ubuntu@192.168.0.210 'cat /etc/rancher/k3s/k3s.yaml' \
  | sed 's/127.0.0.1/192.168.0.210/g' > ~/.kube/k3s-config
export KUBECONFIG=~/.kube/k3s-config
```

## When Adding New Resources

- Add VM resources in `k3s-cluster.tf` following the existing pattern.
- Use `var.k3s_datastore_id` and `var.k3s_network_bridge` — don't hardcode storage or network values.
- Always add `lifecycle { ignore_changes = [disk[0].import_from] }` to VM resources that import disk images.
- Tag all resources with `terraform` tag for identification.
- Use cloud-init for all VM initialization (packages, users, SSH keys).
